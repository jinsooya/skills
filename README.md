# 📁 스킬(SKILL) 작성 규칙 정리

---

## 1. 필수 파일

- 반드시 **`SKILL.md`** 라는 이름의 마크다운 파일이 있어야 함
- 파일 상단에 **YAML 형식**으로 메타데이터 포함 필수

---

## 2. YAML 필수 항목 (Required Fields)

```yaml
name: 스킬-이름
description: 스킬 설명
```

| 항목 | 제약 조건 |
|------|----------|
| `name` | - 최대 **64자**<br>- 소문자, 숫자, 하이픈만 허용<br>- 하이픈으로 시작하거나 끝낼 수 없음<br>- **반드시 부모 디렉토리 이름과 일치**해야 함<br>- 권장 형식: **동명사(verb + -ing)** 형태 |
| `description` | - 최대 **1024자**<br>- 비어있을 수 없음<br>- 스킬이 **무엇을 하는지 + 언제 사용하는지** 함께 기술<br>- 에이전트가 관련 작업을 인식할 수 있도록 **특정 키워드 포함** |

---

## 3. YAML 선택 항목 (Optional Fields)

| 항목 | 설명 |
|------|------|
| `license` | 라이선스 이름 또는 라이선스 파일 참조 경로 |
| `compatibility` | 최대 500자, 환경 요구사항 명시 |
| `metadata` | 임의의 key-value 쌍 |
| `allowed-tools` | 사전 승인된 툴 목록, 공백으로 구분 (실험적 기능, Experimental) |

---

## 4. 폴더 명명(naming) 규칙

- 소문자만 사용
- 단어 사이는 **하이픈(`-`)** 으로 구분
- `claude`, `anthropic` 같은 예약어 사용 금지
- **폴더 이름 = YAML의 `name` 값** 과 반드시 일치

**예시:**
```
assignment-generator  (O)
Assignment_Generator  (X)
claude-skill          (X)
```

---

## 5. 폴더 구조

```
스킬폴더/
├── SKILL.md                        ← 최상위에 위치
├── references/                     ← 외부 참조 마크다운 파일
│   └── assignment-description-template.md
├── scripts/                        ← 실행 가능한 코드/스크립트 파일
│   └── process_data.py
└── assets/                         ← 아이콘, 이미지 등 기타 자산
    |── header.png
    |── icons/
    └── templates/ 
      |── assignments.html
      └── layout.docx
```

- 외부 참조 파일은 반드시 **`references/`** 폴더 안에 위치
- SKILL.md에서 참조 시 `references/파일명` 또는 `scripts/파일명` 형식으로 링크
- 스크립트는 컨텍스트 윈도우 밖에서 별도로 실행되므로 토큰 소비 없음


| 폴더 | 용도 |
|------|------|
| `references/` | 추가 마크다운 문서, 규칙 파일 등 텍스트 참조 자료 |
| `scripts/` | 실행이 필요한 코드 파일 (예: PDF 변환, 데이터 처리 등) |
| `assets/` | 아이콘, 이미지 등 커스텀 강의/스타일 자산 |


### 각 폴더 상세 규칙

#### `/scripts`
- 의존성(dependencies)을 명확하게 문서화
- 스크립트 자체에 명확한 문서 포함
- 에러 핸들링을 명시적으로 작성
- **Note**: 스크립트를 실행할지, 참조용으로만 읽을지 SKILL.md에 명확히 지정

#### `/references`
- 에이전트가 필요할 때 읽을 수 있는 추가 문서 포함
- 각 참조 파일은 하나의 주제에 집중
- **Note**: 100줄 이상의 참조 파일은 상단에 **목차(Table of Contents)** 포함 -> ToC는 에이전트가 전체 범위를 파악할 수 있는데 도움이 됨

#### `/assets`
- **Templates**: 문서 템플릿, 설정 템플릿
- **Images**: 다이어그램, 로고
- **Data files**: 룩업 테이블, 스키마

## 6. Body Content (SKILL.md 본문) 작성 규칙

형식 제한은 없으나, 아래 구성을 권장한다.

**권장 섹션 구성:**
- Step-by-step 단계별 지침
- 입력 형식(Input Format), 출력 형식(Output Format), 입출력 예시
- 자주 발생하는 엣지 케이스(edge cases)

**실용적 가이드라인:**
- **500줄 이내**로 유지
- 상세한 참조 자료는 별도 파일로 분리: 기본 내용만 본문에, 고급/도메인별 내용은 외부 파일 링크
- 참조 파일은 SKILL.md 기준 **1단계 깊이**만 허용 (중첩 파일 참조 금지)
- 명확하고 간결하게, 일관된 용어 사용
- 파일 경로는 **Windows에서도 슬래시(`/`) 사용**

## 7. Progressive Disclosure (점진적 공개)

스킬은 **필요한 정보만 단계적으로 로드**하는 Progressive Disclosure 원칙을 따른다.

> 💡 컨텍스트 윈도우는 공공재(public good)처럼 생각하자.
> 데이터가 많을수록 토큰 소비가 늘고, 컨텍스트 저하나 오류 응답 가능성이 높아진다.

### 로딩 단계

| 단계 | 로드되는 내용 | 설명 |
|------|-------------|------|
| **1단계** | `name` + `description` | 항상 컨텍스트에 포함. Claude가 스킬 인식·트리거 판단에 사용 |
| **2단계** | `SKILL.md` 전체 내용 | 스킬이 트리거된 후 로드 |
| **3단계** | `references/`, `scripts/` 등 | 실제로 필요할 때만 조건부 로드 |

### 핵심 원칙

- 모든 정보를 한꺼번에 로드하지 말고, **필요할 때만** 참조 파일을 읽도록 조건부로 설정
  + 예) '과제 설명서를 작성할 때만 해당 파일 읽기'
- 스크립트는 **컨텍스트 윈도우 밖**에서 별도 실행 → 불필요한 토큰 소비 없음
- 수백 개의 스킬이 있어도, 실제 사용되는 스킬의 내용만 컨텍스트에 올라옴

## 8. 호환성

- 스킬은 **오픈 스탠다드(Open Standard)** 기반
- Claude 외에도 **Codex**, **Gemini CLI**, **Claude Code**, **Open Code** 등 다른 AI 환경에서도 사용 가능
- 팀원과 공유 및 재사용 용이
- 현재 스펙은 활발히 발전 중 → 일부 기존 스킬은 최신 표준을 따르지 않을 수 있음

---

## 9. 언제 스킬을 만들어야 할까?

- 동일한 프롬프트를 여러 대화에서 반복 입력하고 있을 때
- 특정 분석 워크플로우를 매주/매번 반복할 때
- 팀원과 동일한 작업 방식을 공유하고 싶을 때
- Claude가 일반적인 방법은 알지만, **우리 팀/회사 방식**으로 수행해야 할 때
